Process Map & Codebase Summary

Prepared June 19, 2026  ·  Reflects production codebase post PR #4 merge  ·  Confidential — Samawy Internal

1 What the System Does

The Audiobooks App is an internal operations platform built by Samawy for ingesting, processing, and cataloging Arabic audiobooks. It manages the complete pipeline from receiving raw audio from publishers through to producing a publication-ready dossier.

There are three user-facing portals:

Operator Portal
Internal Samawy staff — the primary application. Manages all batch ingestion, book processing, dossier generation, and catalog sync.
Studio Portal
Publishing partners upload audio samples and production files, receive status updates, and request Google Drive file deliveries.
Acquisition Portal
Partner acquisition teams review and approve production files submitted by studios.
Audience
Arabic-language audiobook market. All content, metadata, and email templates are bilingual (Arabic primary, English secondary).

5 End-to-End Process Map

0
Delivery

File Delivery

A publisher sends audio files either by sharing a Google Drive folder or uploading a ZIP. An operator creates a batch in the system recording the source type and entering state ingested. No processing begins until the operator explicitly starts intake.

1
Queue: drive-intake / upload-intake

Intake Normalization

The intake worker lists the Drive folder or extracts the ZIP manifest. It infers the delivery structure (single book ZIP, multi-book batch, flat tracks, subfolders, or mixed). Files are grouped by inferred book — audio files, cover image, and metadata sheet identified per group. Source files are copied from Drive/ZIP to R2 at ingestions/{batchId}/source/. The normalized group structure is saved to batch.normalization.groups.

Resilience: Each file copy is checkpointed. On queue timeout (5 s), an IntakeCheckpointError requeues without counting toward the retry limit. Up to 20 redeliveries allowed; the 8th real failure moves the batch to intake_failed. Operators can manually retry a stuck batch.

→ Status: normalized
2
Queue: metadata-parse

Metadata Parsing

The metadata spreadsheet (XLSX) identified within the batch is loaded. Cloudflare AI (Claude) or a heuristic column detector finds the header row and maps columns to fields: title, author, publisher, ISBN, narrator, blurb, genre, year, pricing, track count. Rows are extracted and saved to batch.normalization.metadataRows.

Operators can re-select the sheet and re-parse with different column overrides if the AI mapping is incorrect.

→ Status: metadata_parsed
3
Manual — Operator

Seller Lock

Operator selects the seller and publisher from a Samawy catalog lookup. This locks the batch commercially and governs catalog association. Currently batch-level only (per-book seller assignment is a planned future feature).

→ Status: seller_locked
4
Manual — Operator

Reconciliation

For each normalized group, an ingestion_candidate record is created. The Samawy catalog is searched for title matches; potential matches are stored with confidence scores. The operator reviews each candidate and sets a decision:

  • Approved (existing) — matches an existing catalog entry
  • Approved (new) — new title to be added
  • Parked — missing files, missing metadata, or needs business review
  • Excluded — duplicate, extra source, or non-book artifact

On finalization: audiobook_record and track_record rows are created for each approved candidate. Metadata precedence: manual override > metadata sheet > catalog candidate.

→ Status: records_created
5
Manual — Operator

Track Preparation & Approval

Prepare: The source ZIP/files are read, filtered by audio extension (mp3, m4a, m4b, wav, flac, aac, ogg), sorted by filename/order, and a TrackDraft[] is built with proposed titles derived from the metadata sheet, audio tags, or filename heuristics.

Approve: The operator reviews and edits track titles and order, then confirms. Each track records its titleProvenance (metadata_sheet / audio_tag / filename / generated_placeholder).

→ Track approval_status: approved
6
Cloudflare Workflow: ProcessingWorkflow

Audio Processing

Operator clicks "Start Processing". The system:

  1. Generates HMAC-signed download + upload URLs for each track (1-hour expiry)
  2. Starts or reuses a container instance for this book
  3. Container downloads each source track (direct audio or extracts from ZIP), probes with ffprobe, transcodes to MP3 at target bitrate, and uploads via multipart upload (25 MB chunks) to artifacts/processed/
  4. Worker polls the container every 2 seconds for up to 30 minutes

On success: book processingStatus = succeeded → DossierWorkflow is triggered automatically.
On failure: retried up to 2 times (10-second delay), then processingStatus = failed.

Reupload flow: If processing fails due to bad source files, the operator can replace the source ZIP in-place (same book ID) and re-trigger processing.

→ processingStatus: succeeded
7
Cloudflare Workflow: DossierWorkflow

Dossier Generation

Triggered automatically after processing succeeds. Zero retries — any failure requires manual reset and re-trigger.

  1. Build XLSX workbook: Sheet 1 = full metadata + track specs (text fields truncated at 32,767 chars for Excel compatibility); Sheet 2 = sample link
  2. Generate audio sample clip via container (if sample_*_seconds configured)
  3. Build dossier audio ZIP of all final MP3 tracks (multipart upload for large books)
  4. Upload workbook + ZIP to dossier/ prefix in R2
  5. Mark dossierStatus = ready
→ dossierStatus: ready
8
Manual — Operator (or automatic)

ClickUp Sync

Triggered manually per book or automatically after dossier finalization:

  1. Map audiobook fields to ClickUp custom fields (configured in Settings UI)
  2. Create or update task in the configured ClickUp list
  3. Attach cover image if available
  4. Generate task description from template (includes workbook download URL, audio ZIP URL, app deep link)
→ clickupSyncStatus: synced
9
Parallel Track — Studio Portal

Studio Integration

Studios interact via a separate magic-link portal (no password — 24-hour token):

  • Upload audio samples for quality review → operator approves/rejects → studio receives email notification
  • Request files to be delivered to their Google Drive folder → background queue syncs → confirmation email sent
  • Download production files prepared by operators

Upload Strategy for Large Files

  • Files ≤ ~25 MB: single PUT to signed URL
  • Files > 25 MB: multipart — POST /api/internal/multipart-start → PUT each 25 MB chunk → POST /api/internal/multipart-complete

Limits: max book size 1 GB; max single track 200 MB. Container↔Worker communication is HMAC-signed (internal secret) to prevent unauthorized access to R2 artifacts.

12 External Integrations

Google Drive API v3

Auth: Service account (JWT token exchange). Used to list Drive folders, stream source files into R2, and push synced studio files to their Drive folder. Retry: 4 attempts with exponential backoff on 408/429/500–504.

Samawy Catalog Proxy

Auth: OAuth2 client credentials. Used during reconciliation to search for existing book titles, list publishers, and look up sellers. Optional — the system degrades gracefully if the proxy is unreachable.

ClickUp API v2

Auth: API token (stored in app_settings, configurable via Settings UI). Used to create/update book tasks, populate custom field values, and attach cover images. Field mappings are fully configurable — no hardcoded field IDs.

Cloudflare AI (Claude)

Used for metadata spreadsheet column detection — identifies which XLSX columns map to title, author, ISBN, etc. Falls back to heuristic regex-based detection on AI failure.

14 Known Constraints & Technical Debt

🔴 Single-region D1 database
D1 is single-replica SQLite. No automatic failover. Limited concurrent write throughput. If Samawy scales to high batch volume this will become a bottleneck and a reliability risk.
🔴 No staging environment
All deployment and validation happens directly in production. There is no formal staging pipeline, feature flags, or canary mechanism.
🔴 Dossier workflow: zero retries
DossierWorkflow is configured with retries: { limit: 0 }. Any transient failure (network blip, AI timeout) requires a full manual reset and re-trigger. Partial dossier state is not resumable.
🟡 Container concurrency cap
Hard limit of 12 concurrent container instances. Under high load, processing jobs queue and operators see no visible indication of queue depth.
🟡 Seller is batch-level only
Seller assignment is per-batch. Assigning a different seller to individual books within a batch is architecturally planned but not yet implemented — requires a DB migration and UI changes.
🟡 ClickUp coupling is tight
ClickUp sync is wired directly into the dossier workflow. Adding a second downstream system (e.g. Notion, Jira) would require code changes rather than configuration. No event-sourcing or webhook pattern.
🟡 Intake queue opacity
The checkpoint loop pattern works but is invisible to operators. There is no monitoring of queue depth, checkpoint iteration count, or estimated completion time for large Drive folders.
🟢 AI metadata parsing variance
Claude is used for spreadsheet column detection with no fine-tuning or feedback loop. Accuracy on unusual publisher spreadsheet formats varies and requires operator manual re-mapping.
🟢 Email delivery limits undocumented
Cloudflare Email Routing rate limits are not documented internally. High-volume batch completions could trigger throttling with no alerting.
🟢 Pipeline.ts monolith
At ~2,400 lines, pipeline.ts handles normalization, metadata parsing, candidate generation, reconciliation, dossier generation, and ClickUp sync. High cognitive load for new engineers.

15 Recommended Consultant Focus Areas

Priority 1 — Reliability: Address the zero-retry dossier workflow and lack of staging environment before scaling batch volume. A single transient failure in dossier generation currently requires operator intervention every time.
Priority 2 — Scalability ceiling: Assess whether D1 single-replica and the 12-container cap will hold at target volume. Consider Hyperdrive for DB access patterns and evaluate whether the container limit is actually hit in practice (monitoring missing).
Priority 3 — Feature gaps: Per-book seller assignment is commercially important and architecturally close (requires DB column + UI only). Intake progress visibility (queue depth, checkpoint count) would meaningfully reduce operator confusion on large batches.
Priority 4 — Maintainability: Consider splitting pipeline.ts into domain modules (intake, metadata, candidates, dossier, clickup). Evaluate whether ClickUp sync should be decoupled via events to support future downstream systems without code changes.

Questions for the Business

  • What is the target batch volume per week? (D1 and container capacity planning depends on this)
  • Are there plans for additional downstream catalog systems beyond ClickUp?
  • What is the acceptable recovery time objective if the Worker goes down?
  • Is the Samawy catalog proxy always available, or should the system handle catalog outages gracefully?
  • Are there plans to expand beyond Arabic-language content? (RTL assumptions are pervasive in the email and UI templates)
